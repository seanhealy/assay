## Coding Conventions

- **Indentation:** Tabs (not spaces)
- **Line width:** 80 columns
- **Imports:** Auto-organized by Biome
- Prefer named exports over default exports
- Use `function` declarations, not arrow functions
- **Function ordering:** Most important first, helpers last. Exported/primary
  functions at the top, internal helpers below.
- **Variable naming:** Use full words, not abbreviations (e.g., `action` not
  `fn`, `context` not `ctx`).
- **Directory modules:** When a module grows into a directory, use `index.ts` as
  a barrel export only. Implementation lives in a named file.
- **Object/interface properties:** Order by importance, not alphabetically.
  Group related properties. Identifying fields first, core fields next,
  optional/metadata fields last.
- **Use `undefined`, not `null`.** Prefer `undefined` throughout the codebase
  for absent values.
- **Multi-line CLI commands.** When running shell commands with `for` loops,
  pipelines, or multiple statements, format them across multiple lines with
  proper indentation rather than chaining everything on one line with `;`.
  Readability matters even for one-off commands.
- **Prefer the Grep tool over CLI `grep`.** When searching file contents, reach
  for the dedicated Grep tool first it handles permissions cleanly. Use CLI
  `grep` only when piping into another shell command that the Grep tool can't
  express.

## Authoring Shims

Shim files live under `src/shims/{filters,tags}/<name>.ts` and each default-
exports a `ShimFilter` / `ShimTag`. The barrel (`index.ts`) in each directory is
auto-generated by `scripts/generate-shim-barrels.ts` on `prebuild` ‚Äî never edit
those files directly.

Before writing a new shim, look up the official Shopify spec:

```
npm run docs:lookup filter handleize
npm run docs:lookup tag style
```

The output is JSON with a `docsUrl` field, decoded HTML entities, and the
official `summary`, `syntax`, `parameters`, `return_type`, and `examples`. Use
it to confirm:

- The exact set of names to register (some filters have aliases ‚Äî `handleize`
  also registers as `handle`).
- The expected return type and parameter shape.
- Whether the entry is `deprecated`.

If a `vendor/shopify-examples/{filters,tags}/<name>.json` file exists, the
lookup also includes a `renderedExamples` array of `{ code, data, output }`
triples scraped from shopify.dev. Use these to drive shim tests ‚Äî render `code`
against `data` and assert the result matches `output`. Run
`npm run docs:fetchExample <kind> <name>` (or `missing` for everything we
haven't fetched yet) to populate the directory before authoring. The directory
is gitignored ‚Äî we don't redistribute Shopify's docs content ‚Äî so each dev
fetches their own local copy after clone.

Reuse rule: extract impl into `src/shims/{filters,tags}/shared/` only when a
second consumer appears. One-off impls live inline in the shim file.

## Objects

Liquid objects (e.g. `product`, `cart`, `shop`) are _data shapes_, not behavior,
so they aren't shimmed at runtime ‚Äî consumers mock them with TypeScript types in
their tests. We still fetch the raw `.md` reference files
(`vendor/shopify-examples/objects/<name>.md`) but don't parse them into JSON.
Object metadata (properties, types) lives in
`vendor/theme-liquid-docs/data/objects.json` and can drive a future TS-type
generator.

1. Run `npm run verify` to lint, format, and type-check
